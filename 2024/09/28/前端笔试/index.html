前端性能优化

什么是css图层

浏览器在渲染页面时，会将一个页面分成多个图层，每个图层有大有小。一个图层上有一个或者多个节点。在渲染dom的时候，浏览器做的工作实际上是

  • 获取dom后，分割为多个图层
  • 对每个图层的节点计算样式结果（Recalcute style –样式重计算）
  • 为每个节点生成图形和位置（layout-回流和重布局）
  • 将每个节点绘制填充到图形当中（重绘）
  • 图层作为纹理，上传至GPU当中
  • 复合多个图层到页面上生成最终屏幕图像（图层重组）

什么情况下会创建图层

  • 3D或者透视变换，CSS属性
  • 使用加速视频解码的video节点
  • 拥有3D（webGL） 上下文或加速的2D上下文的canvas节点
  • 混合插件（flash）
  • 对自己的opacity做css动画，或使用一个动画webkit变化的元素
  • 拥有加速CSS过滤器的元素
  • 元素有一个包含符合层的后代节点
  • 元素有一个z-index较低且包含一个复合层的兄弟元素

需要注意的是： 如果图层中的某个节点需要重绘，那么整个图层都需呀重绘。比如一个图层包含很多节点，其中有个gif图，gif图的每一帧，都会重绘

最终生成最终的图层位图。 所以这需要通过特殊的方式来强制gif图属于自己的一个图层*translate（0）或者 translate3d（0，0，0），css3的动画也一样

简化一下css动画的生成过程

  • 计算要被加载到节点上的样式结果（ 样式重计算）
  • 为每个节点填充到图层中
  • 组合图层到页面上

触发重布局的属性

有些节点， 当你改变他时， 会需要重新布局，（ 这也意味着，要重新计算其他被影响的节点的位置和大小） 。这样的情况下，被影响的dom树越大，重绘所需要的时间，就会越长。 而渲染一帧动画所需要的时间也响应的越长。所以要尽力避免这些属性

一些常用的改变时会触发重布局的属性

盒子模型相关属性发生变化会触发重布局

  • width
  • height
  • padding
  • margin
  • display
  • border-width
  • border
  • min-height

定位属性及浮动也会触发重布局

  • top
  • bottom
  • left
  • position
  • float
  • clear

改变节点内部文字结构也会触发重布局

  • text-align
  • overflow-y
  • font-weight
  • overflow
  • font-family
  • line-height
  • vertical-align
  • whight-space
  • font-size

这些属性触发重排的特定就是改变节点的大小和位置，所以会触发重排

只触发重绘的属性有

  • color
  • border-style
  • border-radius
  • visibility
  • text-decoration
  • background
  • background-image
  • background-position
  • background-repeat
  • background-size
  • outline-color
  • outline
  • outline-style
  • outline-width
  • box-shaddow

这些属性都不会改变节点的大小和位置， 自然不会触发重布局

注意： 在重绘时，这些节点被加载到gpu中进行重绘，这对移动设备比如手机的影响还是很大的。 因为cpu不如台式机或者笔记本电脑。 所以所消耗的时间更长。

强迫浏览器创建图层

在blink和webKit浏览器中， 当一个节点设定了透明度过度效果或者动画时。浏览器会将其作为一个单独图层。 但是很多开发者使用translateZ(0),或者translate3d(0,0,0) 去使浏览器创建图层。这种方式可以消除动画在开始之前图层创建的时间。使得动画尽快开始（创建图层和绘制图层还是比较慢的） 。而且不会随着抗锯齿而突出。不过这种方法需要节制， 否则会因为创建过多的图层导致崩溃。

left 和top替代品

我们通过transform可以修改节点的位置， 旋转，大小等。 我们通常会使用left，和top属性来修改节点的位置。 但是正如上面所说， left和top会触发重布局。 修改时的代价相当大，取而代之，更好的办法是使用translate

JS动画和CSS3动画的比较

js动画

缺点： 在浏览器主线程中运行，而其中还有很多其他需要运行的js，样式计算，布局，绘制等对其干扰。 这也导致了线程可能出现阻塞， 从而造成丢帧的情况

优点： js动画和css预先定义好的动画不同。 可以在动画过程中对其进行控制： 开始，暂停， 回放，终止，取消都可以做到的。 而且一些动画效果，比如视差滚动效果， 只有js能够完成

css动画

缺点： 缺乏强大的控制能力， 而且很难以有意义的方式结合到一起。使得动画变得复杂而且容易出现问题。

优点： 浏览器可以对动画进行优化， 它必要时可以创建图层。 然后在主线程之外运行

V8引擎的垃圾回收机制

V8引擎存在限制

与其他语言不同的是，node限制了js所能使用的内存， （64位为1.4GB, 32位为0.7gb） ，这就意味着无法直接操作一些大的内存对象

为何限制：

  • 表面上看是由于V8最初是作为浏览器的js引擎涉及的。不太可能遇到大量内存的场景。
  • 而深层次的原因是由于V8垃圾回收机制的限制。由于V8需要保证js应用逻辑和垃圾回收器看到的不一样。
  • V8在执行垃圾回收时会阻塞js应用逻辑，直到垃圾回收结束再重新执行js应用逻辑， 这种行为被称为全停顿（stop-the-world） 。 V8的堆内存为1.5GB。V8做一次小的垃圾回收要50ms以上， 做一次非增量式的垃圾回收甚至要1s以上，这样浏览器1s内失去对用户的响应，造成假死现象。 如果有动画效果的话。 动画的展现也将显著受到影响。

突破限制

这个限制是可以打开的，类似于JVM， 我们通过在启动node时可以传递 —max-old-space-size 或者 –max-new-space-size 来调整 内存限制的大小， 前者确定老生代大小， 单位为MB，后者确定新生代大小，为KB。 这些配置只在V8初始化时生效，一旦生效不能再改变

V8的堆构成

V8的堆其实并不是有老生代和新生代两部分构成的。 可以将堆分成几个不同的区域。

  • 新生代内存区： 大多数的对象被分配在这里，这个区域很小，但是垃圾回收特别频繁
  • 老生代指针区： 属于老生代， 这里包含了大多数可能存存在指向其他对象的指针的对象。 大多数从新生代晋升的对象都会被移动到这里。
  • 老生代数据区： 属于老生代， 这里保存原始数据对象，这些对象没有指向其他对象的指针
  • 大对象区： 这里存放体积超越其他区大小的对象，每个对象都有自己的内存， 垃圾回收不会移动大对象
  • 代码区： 代码对象， 也就是包含JIT之后指令的对象，都会分配在这里。唯一拥有执行权限的内存区
  • cell区，属性cell， map区，属性cell和map。每个区域都存放相同大小的元素，结构简单

每个区域都是由一组内存页组成， 内存页是V8 申请的最小单位， 出来，除了大对象的内存页比较大以外，其他区的内存页都是1MB大小， 而且按照1MB对其的。 内存页除了存储的对象，还由包含一个数据和标识信息的页头。 以及一个用于标记哪些对象是活跃对象的位图区。 另外每个内存页还有一个单独分配在内存区的槽缓冲区， 里面放着一组对象，这些对象可能指向其他存储页的对象。垃圾回收置灰针对新生代内存区， 老生代指针区，以及老生代数据区进行垃圾回收

如何判断回收内容

如何确定哪些内存需要被回收，哪些内存不需要被回收，这是垃圾回收需要解决的最基本的问题。

  • 活对象： 当且仅当它被一个根对象或者活对象指向

  • 根对象永远是活对象， 它是被浏览器或者V8所引用的对象， 被局部变量所指向的对象也属于根对象， 因为他们所在的作用域被视为根对象， 全局对象也属于根对象， 浏览器也属于根对象

如何识别指针和数据

垃圾回收器需要面临一个问题，它需要判断哪些是数据，哪些是指针，， 由于很多垃圾回收算法会将对象在内存中移动。所以经常需要进行指针的改写

三种办法识别指针

  • 保守法

    将所有堆上的对其的字都认为是指针，那么有些数据被误认为是指针， 于是有些实际的数字的假指针， 会被误认为指向活跃对象。导致内存泄漏（假指针指向的对象可能是死对象， 但依旧有指针指向–这个假指针指向它） ，同时我们不能移动任何内存区域

  • 编译器提示法： 如果是静态语言，编译器能够告诉我们每个类中指针具体的位置， 而一旦我们直到对象哪个实例化得到的。就能知道对象所有指针， 这是JVM实现垃圾回收的方式，但这种方式并不适合js这样的动态语言

  • 标记指针法

    这种方法需要在每个字末位预留一位来标记这个字段是指针还是数据。这种方法需要编译器支持，但实现简单，而且性能不错。V8采用的是这种方式。V8将所有数据以32bit字宽来存储，其中最低一位保持0，而且指针最低两位为01

    V8的回收策略

    v8采用了一种分代回收算法，将内存分为两个声代。新生代和老生代。 新生代对象为存活时间较短的对象，老生代中的对象为存活时间较长或常驻内存的对象。 分别对新生代和老生代使用不同的垃圾回收算法来提升垃圾回收效率。对象起初都会被分配到新生代，当新生代中的对象满足某些条件时，会被移动到老生代

    V8的分代内存

    默认情况下，64位环境下的V8引擎的新生代内存大小位32MB，老生代位1.4gb 。 32位则减半

    v8内存的最大保留空间分别位1464MB， 和732MB。 具体的计算公式位： 4*reserved_semispace_space + max_old_generation_size。 新生代有两块reserved_semispace组成。 每块位16MB或者8MB

    新生代的特点

    大多数对象被分配在这里， 这个区域很小但是垃圾回收特别频繁。 在新生代分配内存非常容易。我们只需要保存一个指向内存的指针，不断根据新对象的大小进行递增即可。 当该指针达到了新生代内存区的末尾。 就会有一次清理（ 仅仅时清理新生代）

    新生代垃圾回收： 引用计数法+复制清除

    对象晋升

    当一个对象经过多次新生代的清理依旧幸存，则说明它的生存周期比较长。 也就是会被移动到老生代。这就是对象晋升。

    • 对象从from空间复制到to空间的适合，会检查它的内存地址来判断这个对象是否已经经理过一个新生代的清理。 如果是， 则复制到老生代当中，否则复制到to空间中
    • 当对象从from空间复制到to空间时， 如果to空间已经被使用超过了25%， 那么这个对象直接被复制到老生代

    老生代的垃圾回收算法

    老生代的特点

    老生代所保存的对象大多数时生存周期很长甚至是常驻内存的对象，而且老生代所占用的内存较多

    老生代的垃圾回收算法

    老生代占用的内存比较多。 如果使用scavenge算法。浪费一般空间。 所以采用标记清除+标记整理的算法

    标记清除分为两个阶段： 标记阶段遍历堆中所有对象，标记活着的对象， 在清除阶段，只清除没有被标记的对象。

    问题: 进行一次标记清除后，内存空间不连续， 会出现很多内存碎片，如果碎片不够用。 会导致提前触发垃圾回收
